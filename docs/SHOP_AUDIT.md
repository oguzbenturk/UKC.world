# Shop System Audit — UKC.world
**Date:** March 3, 2026  
**Scope:** Full-stack shop audit — product catalog, cart, checkout, payments, orders, admin

---

## 1. Architecture Overview

```
User browses shop (/shop or /shop/:category)
    │
    ▼
Shop.jsx — single upfront fetch of ALL products → client-side filter/sort
    │
    ▼
ProductCard → click → ProductPreviewModal (select size/color/variant)
    │
    ▼
CartContext (localStorage-persisted) → ShoppingCart drawer
    │
    ▼
CheckoutModal → POST /api/shop-orders
    │
    ├── Wallet payment → walletService.recordTransaction → order confirmed immediately
    └── Credit card → Iyzico redirect → [callback] → order update (⚠️ BUG — see §5)
```

## 2. User Buy Flow (Step-by-Step)

### Step 1 — Browse Products
- Route: `/shop` (landing) or `/shop/:category` (filtered)
- All products are fetched once via `GET /api/products/shop/by-category?limit=100`
- Filtering is 100% client-side (category, subcategory, brand, sort, search, in-stock toggle)
- Category nav pills at top of page + sidebar filters on left
- Products displayed in a 6-column grid (`xl:4` columns per Ant Design Col)

### Step 2 — Open Product Preview
- Click any `ProductCard` → opens `ProductPreviewModal`
- Modal shows: image gallery (multi-image carousel), brand, name, description, price, stock badge
- Supports size / color / variant selection (parsed from `variants`, `colors`, `sizes` JSON fields)
- Auto-selects first available color and size
- Shows variant-specific stock if selected

### Step 3 — Add to Cart
- **Unauthenticated users:** clicking "Add to Cart" triggers `openAuthModal()` — prompts login first ✅
- **Authenticated users:** item added to `CartContext` with a unique `cartItemId` = `{id}-{size}-{color}`
- Same product with same size+color increments quantity; different variant = separate line item
- Cart is persisted in `localStorage` under key `plannivo_cart`
- Floating cart button (bottom-right, fixed) shows badge count

### Step 4 — Manage Cart
- Open cart via floating button → `ShoppingCart` drawer (Ant Design Drawer)
- Per-item: update quantity (+/-), remove, move to wishlist
- Footer shows: subtotal, wallet balance indicator (green if can afford, red if insufficient), dual-currency display
- "Wallet low — other payment options available" tag if balance insufficient
- "Proceed to Checkout" button

### Step 5 — Checkout
- Opens `CheckoutModal`
- **Payment methods:**
  - `wallet` — pay from in-app balance (disabled if insufficient)
  - `credit_card` — redirects to Iyzico payment page
  - `cash` — creates order pending manual confirmation
- **Address:** required for all orders; stored as text in `shipping_address` field
- Displays order summary (item list, quantity, unit price, total)

### Step 6 — Order Creation (Backend)
`POST /api/shop-orders`

```
1. Validate each item — re-reads price from DB (price spoofing protection ✅)
2. Validate stock per item (returns 400 if insufficient)
3. Check wallet balance if method = 'wallet'
4. BEGIN transaction
5. INSERT into shop_orders (status='pending', payment_status='pending')
6. INSERT into shop_order_items (per item)
7. UPDATE products SET stock_quantity = stock_quantity - qty  (atomic ✅)
8. Process payment (wallet deduct OR Iyzico redirect)
9. COMMIT
10. Emit socket event → shop:newOrder (notifies admin in real-time ✅)
11. Emit shop:lowStock if any item falls below threshold
```

**Order number** is auto-generated by DB trigger: `ORD-{YYYYMMDD}-{sequence}`

### Step 7 — Payment Processing

| Method | Flow | Order Status After |
|--------|------|--------------------|
| Wallet | Deducts via `walletService.recordTransaction` synchronously | `confirmed` + `payment_status=completed` ✅ |
| Credit Card (Iyzico) | Redirects to Iyzico hosted page; returns `paymentPageUrl` | `pending` + `payment_status=pending` ⚠️ |
| Cash | Order created only | `pending` + `payment_status=pending` (admin updates manually) |

### Step 8 — Order Confirmation
- **Wallet path:** `CheckoutModal` receives `success: true`, shows order confirmation screen with `order_number`
- **Card path:** User is redirected to Iyzico page. After payment, redirected to `/payment/callback?status=success|failed`
- Cart is cleared on success (`clearCart()`)

## 3. Admin / Staff Order Management

### Admin Views
| Route | File | Description |
|-------|------|-------------|
| Admin panel → Shop Orders | `ShopOrdersPage.jsx` → `OrderManagement.jsx` | Full order list, search, status filters |
| Finance → Shop | `FinanceShop.jsx` | Revenue, margins, order analytics |

### Order Status Lifecycle
```
pending → confirmed → processing → shipped → delivered
              │
              └──→ cancelled (restores stock ✅)
              └──→ refunded  (credits wallet ✅)
```

### Admin Capabilities (all in `PATCH /:id/status`)
- Update any order status with notes
- Timestamps auto-set: `confirmed_at`, `shipped_at`, `delivered_at`, `cancelled_at`
- Cancellation: stock automatically restored for all order items ✅
- Refund: wallet credit issued automatically if `payment_status = 'completed'` ✅
- Full status history logged in `shop_order_status_history` ✅

### Quick Sale (Staff / Front Desk)
- `POST /api/shop-orders/admin/quick-sale`
- Accepts `user_id` (optional — null for walk-in customers)
- Supports `cash`, `card`, `wallet` payment methods
- Order auto-created as `confirmed` + `payment_status=completed`
- Wallet deduction handled if `payment_method = 'wallet'`

### Low Stock Monitoring
- Auto-detected after every order: checks if stock ≤ `low_stock_threshold` (default 5)
- Emits `shop:lowStock` socket event to admin
- Dedicated endpoint: `GET /api/shop-orders/admin/low-stock`

## 4. Data Layer

### Tables
| Table | Purpose |
|-------|---------|
| `products` | Product catalog (name, brand, price, stock, category, subcategory, images, variants as JSON) |
| `product_subcategories` | Subcategory metadata (display_name, display_order, parent_subcategory) |
| `shop_orders` | Order header (user, status, payment, totals, shipping_address, timestamps) |
| `shop_order_items` | Order line items (product snapshot at time of order, size/color/variant) |
| `shop_order_status_history` | Audit trail of all status changes |

### Key Design Decisions
- **Price snapshot:** `unit_price` in `shop_order_items` is copied from `products.price` at order time — correct ✅
- **Product snapshot:** `product_name`, `product_image`, `brand` are copied — orders survive product deletion ✅
- **Stock validation:** server-side re-check on checkout — cart quantity is advisory only ✅
- **order_number:** DB trigger generates `ORD-YYYYMMDD-NNNNNN` automatically ✅
- **Currency:** hardcoded `EUR` throughout (no multi-currency checkout) — uses `CurrencyContext` for display only

## 5. Bugs & Issues Found

### 🔴 CRITICAL — Iyzico Credit Card Callback Does Not Update Shop Orders
**Location:** `backend/server.js` Iyzico callback handler  
**Problem:** The callback at `POST /api/finances/callback/iyzico` only looks for **wallet deposit requests** (`approveDepositFromCallback`). When a shop order is paid by credit card, Iyzico calls back with a token but the handler has no matching deposit request → the callback fails silently. The shop order remains at `payment_status = 'pending'` forever, even after the customer has paid.  
**Impact:** Every card payment for a shop order is effectively lost — the stock is decremented, the order exists, but it never gets confirmed. Admin must manually update.  
**Fix needed:** In the Iyzico callback, detect if the `referenceCode` is a shop order number (`ORD-*`) and update `shop_orders.payment_status = 'completed'` + `status = 'confirmed'` accordingly.

---

### 🟠 HIGH — No Customer Order History Page
**Problem:** `GET /api/shop-orders/my-orders` exists in the backend but there is no frontend page where customers can view their own past orders.  
**Impact:** Customers have no self-service order visibility. If they close the checkout confirmation, the only record is the order number they saw on screen.  
**Fix needed:** Add an "My Orders" tab or section to the student/customer dashboard using `GET /api/shop-orders/my-orders`.

---

### 🟠 HIGH — Product Images 404 in Local Dev
**Location:** `backend/uploads/images/`  
**Problem:** Products in the database reference image paths like `/uploads/images/image-{uuid}-{timestamp}.png`. These files only exist on the production server. In local development, all product images return 404.  
**Impact:** Product cards and preview modals show no images during local development.  
**Fix needed:** Either sync the uploads folder from production, or use placeholder images for dev, or configure a CDN base URL so image paths resolve to production.

---

### 🟡 MEDIUM — Cash Payment Orders Never Auto-Confirmed
**Problem:** When `payment_method = 'cash'`, the order is created with `status = 'pending'` and `payment_status = 'pending'`. There is no mechanism to mark the order or notify admin that cash is expected at pickup.  
**Impact:** Admin must know to look for cash orders and manually confirm them.  
**Fix needed:** Either: (a) emit a specific socket event `shop:cashOrderPending` for admin notification, or (b) set cash orders to `status = 'confirmed'` and `payment_status = 'pending'` immediately so they appear in the fulfillment queue.

---

### 🟡 MEDIUM — Cart Stores Stale Prices from localStorage
**Problem:** The cart is persisted in localStorage. If a product's price is updated by an admin between cart add and checkout, the displayed cart total is wrong. The server correctly re-reads prices from DB on order creation, so the actual charge is correct — but the user's displayed total may differ from what they are charged.  
**Impact:** Small trust issue. User sees €X in cart, pays €Y (different amount) on confirmation.  
**Fix needed:** On cart open, optionally re-validate cart item prices against the backend, or show a disclaimer that prices may change.

---

### 🟡 MEDIUM — No Discount / Promo Code System
**Problem:** The `shop_orders` table has a `discount_amount` column but there is no frontend UI or backend logic for applying discount/promo codes.  
**Impact:** The field exists but is always `0`. If discounts are planned, the plumbing is there but the feature is not implemented.

---

### 🟢 LOW — No Shipping Cost Logic
**Problem:** `totalAmount = subtotal` — no shipping fees. Line in code has comment: `// Can add tax/shipping logic here`.  
**Impact:** All orders are free shipping. Fine if that's intentional (in-store pickup / digital goods), but may need to be addressed for physical product delivery.

---

### 🟢 LOW — `order_number` Not in `shop_orders` INSERT
**Problem:** (Non-breaking) The INSERT into `shop_orders` omits `order_number` from the column list. A DB trigger auto-generates it. If the trigger is ever dropped or the column constraint changes, the INSERT would fail.  
**Impact:** Currently works. Minor code clarity issue.  
**Fix:** Explicitly generate and pass `order_number` in the INSERT, or document the trigger dependency.
